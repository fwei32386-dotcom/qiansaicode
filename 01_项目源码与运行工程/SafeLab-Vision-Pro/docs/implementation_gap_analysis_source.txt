1. 文档使用方式与总原则
本文档是 SafeLab-Vision Pro 项目的共同工作契约，用于三名成员并行开发、统一接口、交给各自 Codex 作为上下文背景和任务命令。
它不是 PPT，也不是答辩稿；它是研发执行文档。三个人每次开启新的 Codex 对话时，应先提供“全员统一上下文”，再提供本人角色章节和当前任务卡。
1.1 一句话原则
不要把系统做成普通 YOLO Demo。 目标是做成基于 RK3588 的实验室视觉风险认知与边缘自治平台： 视觉输入 -> 低延迟推理 -> 场景语义理解 -> 风险规则编排 -> 事件状态机 -> 可解释告警 -> 证据链复盘 -> 自动评测。
1.2 全员硬约束
先冻结统一接口，再分别开发；任何字段新增、命名变更、含义调整，必须先更新 interface_spec.md。
三人不能互相等待真实硬件或真实模型；必须用模拟数据、样例视频、假 Detection、假 RiskEvent 先打通闭环。
所有慢操作异步化：日志、截图、语音、Web 推送不得阻塞主推理链路。
所有队列默认 maxsize=1 或 latest-only，旧帧直接覆盖，禁止旧帧堆积造成累计延迟。
PPT 与答辩材料不纳入三人研发分工；三人的交付只按代码、配置、测试、日志、可运行系统验收。
每个模块必须有输入、输出、样例、验收命令、失败回退方式。

3.2 主数据流
VideoSource(camera/hdmi/file) -> VideoFrame(frame_id, source_type, timestamp, frame) -> FrameScheduler(select source / decide full-frame or ROI / decide detection frequency) -> RKNNDetector -> Detection[] -> TrackManager(PersonTrack / FireSmokeTrack) -> RuleEngine(DSL rules + semantic map) -> RiskEvent -> EventStateMachine -> AlarmAction -> TimelineLogger + SnapshotManager + SQLite + Dashboard
3.3 低延迟原则
采集线程只维护最新帧，旧帧直接覆盖。
显示线程独立运行，不等待推理结果。
推理线程只处理当前最新帧或调度器选中的 ROI。
告警、截图、日志、Web 推送全部走异步事件队列。
同一事件只在状态变化时执行重动作，避免每帧重复报警。
普通状态低频全帧检测，疑似风险状态提高相关目标检测频率。
双路输入时默认双路显示、单路推理，避免 NPU 和内存带宽被压满。
