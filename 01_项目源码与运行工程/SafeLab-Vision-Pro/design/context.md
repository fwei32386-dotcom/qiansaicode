# SafeLab-Vision Pro Web UI Design Context

## Project
SafeLab-Vision Pro is an RK3588 edge-vision safety demo. The web entry point is `reports/index.html`, generated by `tools/generate_report_index.py`, and served locally from the pulled board report package.

## Audience
- Primary: teachers, reviewers, and demo viewers who need to understand the system in under one minute.
- Secondary: the builder/operator who needs fast access to reports, board evidence, and debug dashboards.

## Primary Jobs To Be Done
- Show that the board is connected and the demo evidence is real.
- Make the first viewport explain the product story: input source -> vision analysis -> risk decision -> alarm action -> evidence report.
- Surface the active input source clearly, including Camera and HDMI availability.
- Make the camera preview visually prominent without hiding status and completion evidence.
- Preserve engineering access to reports, acceptance evidence, and live dashboard links.

## Design Goal
Create a hybrid "acceptance showcase + product demo home" for SafeLab. The page should feel like a polished safety-operations product, not a raw report index. It should be impressive enough for demo review while still honest about current board/runtime state.

## Information Hierarchy
1. Product identity and board/demo status.
2. Input source selector/status: Camera, HDMI, selected source, and fallback when state data is missing.
3. Live visual evidence: board camera snapshot.
4. Risk and operational metrics: risk state, events, high-risk count, actions, full-system completion.
5. System story/flow: input source, RKNN/vision, rule engine, alarm action, evidence reports.
6. Primary actions: Live dashboard, alarm review, final acceptance, export package.
7. Report library and raw evidence files.

## Required States And Interactions
- Current input source must never appear as unexplained `unknown` when camera evidence exists; default to Camera with a "state not reported" detail if missing.
- Camera and HDMI must be visible as source modes. The selected mode must be visually distinct.
- Missing or incomplete hardware status must be shown honestly but not dominate the first impression.
- Links to live dashboard, preview report, final acceptance, and export must remain accessible.
- The page must work as static HTML served from `reports/`, with optional JSON refresh from `live_dashboard_state.json` and `final_acceptance_summary.json`.

## Visual Direction Constraints
- Use an industrial, safety-control-room aesthetic: dark operational background, restrained gold/cyan accents, crisp borders, dense but readable cards.
- Avoid a generic card wall. The first viewport should have a memorable composition with a product story.
- Do not use decorative orbs or generic purple/blue gradients.
- Keep card radius at 8px or less.
- Text must not overflow on desktop or mobile.

## Technical Constraints
- Source of truth is `tools/generate_report_index.py`; generated `reports/index.html` must not be hand-edited as the only change.
- Existing report files and JSON data are local/offline artifacts.
- Browser verification should use the in-app browser at `http://127.0.0.1:8766/index.html`.
- Add or update tests in `tests/test_report_index.py` for durable UI requirements.

## Success Criteria
- First viewport clearly communicates "SafeLab safety vision demo running on board evidence."
- Input source is visible and understandable without opening another dashboard.
- Camera snapshot, risk state, completion percentage, and core metrics are visible above the fold on desktop.
- A reviewer can follow the system story from source to evidence in one glance.
- Engineering report access remains one click away.
- `python -m unittest tests.test_report_index` passes.
- Browser QA shows no console error/warning and confirms required UI elements are visible.

## Open Questions Or Assumptions
- Assumption: the preferred direction is a hybrid of "competition/acceptance showcase" and "productized demo home".
- Assumption: when `live_dashboard_state.json` lacks `input_source`, the page should infer Camera from camera evidence and show that it is inferred.
- Open question: whether the page should include a real live preview link to `start_board_camera_live_preview.ps1` output, or only the pulled snapshot.

## Approved Direction
Mission Playback acceptance cockpit. The first viewport should read like a working safety mission: a strong board/demo status header, a visible source selector, a camera evidence stage, and a horizontal pipeline showing `Camera/HDMI -> RKNN Vision -> Rule Engine -> Alarm Action -> Evidence Pack`. The design should make the reviewer understand the system path before they inspect the report library.
